Next Task: Port kernel_mul_mm_id (mat-mat with expert index) for MoE

r2 uses kernel_mul_mm_id with simdgroup_half8x8 (64×32 tile SGEMM) instead of per-expert mat-vec. This wins for prefill (batch > 1) and speculative decoding. For single-token decode, both r2 and l26f use mat-vec.

Key insights for restarting

kernel_mul_mm_id uses simdgroup_half8x8 with 64×32 output tiles, 4 SIMD groups, 128 threads per threadgroup, 8KB threadgroup memory
For single-token decode mat-vec is optimal. mat-mat wins when batch ≥ 32 tokens share an expert
The 1.56x decode gap is kernel quality + dispatch efficiency, not mat-vec vs mat-mat
Use ds4_metal_args_mul_mv (from dense.metal) as the Metal-side struct — NOT ds4_metal_q8_0_matvec_args (host-only)
Add #ifndef QK_K / #define QK_K 256 / #endif guard in l26f_dense.metal for cross-file macro safety
The MoE weight tensors (ffn_gate_exps, etc.) are stored contiguously as 3D [256, out_dim, in_dim] — nb02 stride works directly
